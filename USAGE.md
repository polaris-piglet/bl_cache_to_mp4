# bl_cache_to_mp4 使用指南

## 完整参数说明

| 参数 | 缩写 | 类型 | 默认值 | 说明 |
|------|------|------|--------|------|
| `--input` | `-i` | 路径 | **必填** | Bilibili 缓存目录路径 |
| `--output` | `-o` | 路径 | **必填** | MP4 输出目录（不存在会自动创建） |
| `--platform` | `-p` | 选项 | 自动检测 | 缓存格式：`android` / `windows` / `mac` |
| `--ffmpeg` | `-f` | 路径 | 自动查找 | ffmpeg 可执行文件路径 |
| `--jobs` | `-j` | 整数 | 2 | 并发合并线程数 |
| `--danmaku` | — | 开关 | 关 | 同时导出弹幕 XML 文件 |
| `--move-completed` | — | 路径 | 无 | 合并成功后将文件移动到此目录 |
| `--single-output` | — | 开关 | 关 | 所有文件输出到同一目录（不按合集分子文件夹） |
| `--add-index` | — | 开关 | 关 | 文件名前添加分P序号（如 `1.标题.mp4`） |
| `--delete-source` | — | 开关 | 关 | 转换成功后删除源缓存文件夹（与 `--move-completed` 互斥） |
| `--export-cover` | — | 开关 | 关 | 导出封面图片为独立 .jpg 文件 |
| `--export-info` | — | 开关 | 关 | 导出原始元数据为 .info.json 文件 |

> **互斥关系**：`--move-completed` 和 `--delete-source` 不能同时使用。
> - `--move-completed`：转完后把输出文件搬到完成目录
> - `--delete-source`：转完后删除源缓存文件夹
> - 都不加：转完后什么都不动

## 视频元数据

脚本**默认自动嵌入**以下元数据到 MP4 文件中（无需额外参数）：

| MP4 标签 | 内容来源 | 说明 |
|----------|---------|------|
| `title` | 视频标题 | 播放器和文件管理器中显示 |
| `artist` | UP主名 | Android 缓存的 `owner_name` 字段 |
| `album` | 合集名 | 方便按合集归类 |
| `track` | 分P号 | 播放器中显示集数 |
| `comment` | BV/AV号 | 可追溯到原始视频 |
| 封面图 | cover.jpg | 文件管理器缩略图（如果缓存中有封面图） |

这些都是 MP4 容器标准字段，Windows 资源管理器右键属性可看到，主流播放器都识别。

**无法嵌入 MP4 的信息**需要用独立文件导出：

| 信息 | 导出方式 | 参数 |
|------|---------|------|
| 弹幕 | .xml 文件 | `--danmaku` |
| 封面原图 | .jpg 文件 | `--export-cover` |
| 完整原始元数据 | .info.json 文件 | `--export-info` |

## 重复文件处理

如果输出目录中已存在同名 MP4 文件，脚本会在确认合并后提示选择：

```
检测到 5 个视频在输出目录中已存在:
  - 合集名/标题1.mp4
  - 合集名/标题2.mp4
  ...

请选择处理方式:
  1. 跳过重复 — 已存在的不再转换
  2. 覆盖重复 — 重新转换并覆盖已有文件
  3. 生成副本 — 已存在时自动添加编号后缀 (默认)
```

- **选 1（跳过）**：节省时间，适合增量转换
- **选 2（覆盖）**：重新转换并替换已有文件
- **选 3（副本）**：生成 `标题(0).mp4`、`标题(1).mp4`

没有重复文件时不会出现此提示。

## 常用场景

### 场景 1：最简单 — 只转换

```sh
python bl_cache_to_mp4.py -i D:\bilibili_cache -o D:\output
```

**效果：** 自动检测格式，2线程，MP4 自动嵌入元数据，按合集分子文件夹输出，源缓存不动。

**适合：** 第一次使用，先看看效果。

---

### 场景 2：完整归档 — 转换 + 保留全部信息

```sh
python bl_cache_to_mp4.py -i E:\videos -o E:\video_output -p android -j 4 --danmaku --export-info
```

**效果：** 每个视频输出 3 个文件：`标题.mp4`（内嵌元数据）+ `标题.xml`（弹幕）+ `标题.info.json`（原始元数据），源缓存不变。

**适合：** 完整保存所有信息，之后手动决定是否删除源文件。

---

### 场景 3：转换后清理 — 边转边删

```sh
python bl_cache_to_mp4.py -i E:\videos -o E:\video_output -p android -j 4 --danmaku --delete-source --export-info
```

**效果：** 在场景 2 基础上，转换成功后立即删除该视频的整个源缓存文件夹。失败的保留。合集文件夹变空后也会自动删除。

**适合：** 磁盘空间紧张，确认转换质量没问题后使用。

**风险：** 删除不可恢复，建议先用场景 2 测试。

---

### 场景 4：增量转换 — 跑过一次，后续只转新增的

```sh
python bl_cache_to_mp4.py -i D:\cache -o D:\output -j 4 --danmaku
```

第二次运行时检测到重复，输入 `1` 跳过已存在的，只转换新增视频。

**适合：** 定期缓存新视频，每次只转增量部分。

---

### 场景 5：转换后搬运 — 成功的搬走，失败的留下

```sh
python bl_cache_to_mp4.py -i D:\cache -o D:\temp_output --move-completed D:\final_output -j 4 --danmaku
```

**效果：** 先转到临时目录，成功后移动到最终目录。失败的留在临时目录方便排查。

**注意：** 不能和 `--delete-source` 同时用。

---

### 场景 6：全部导出到同一目录，加序号

```sh
python bl_cache_to_mp4.py -i D:\cache -o D:\output --single-output --add-index
```

**效果：** 不分子文件夹，所有 MP4 平铺输出。文件名前加分P序号：`1.标题.mp4`。

---

## 风险提示

> 请在使用前仔细阅读本节。

### 1. `--delete-source`：永久删除源缓存文件夹

**最高风险操作。** 转换成功后立即删除该视频的整个源缓存文件夹（包括所有画质、元数据、弹幕、封面）。

- 使用 `shutil.rmtree` 直接删除，**不经过回收站，不可恢复**
- 转换失败的不会删除
- 建议：先不加此参数测试几个视频，确认质量后再使用

### 2. `--move-completed`：文件会被移走

合并成功的 MP4（和 sidecar 文件）从输出目录**移动**到完成目录，不是复制。移动后输出目录中该文件不再存在。

### 3. 解密产生的临时文件

合并 Windows/Mac 缓存时，会在源文件旁边生成 `.hlb_temp.mp3` / `.hlb_temp.mp4` 临时文件。正常情况合并后自动删除。脚本崩溃时可能残留，手动搜索删除 `*.hlb_temp.*` 即可。

### 4. 合并失败时输出文件自动删除

ffmpeg 合并失败的不完整 MP4 会被自动删除，避免损坏文件污染输出目录。源缓存不受影响。

### 5. 输出文件不会覆盖

已存在同名文件时自动添加 `(0)`、`(1)` 后缀（除非选择了覆盖模式）。

### 6. 网络路径扫描极慢

NAS/网络共享路径扫描非常慢（实测 6000 个视频约 28 分钟）。建议先拷贝到本地。

### 7. 权限不足的目录静默跳过

没有读取权限的子目录会被跳过，不报错。查看终端 `[警告]` 信息确认。

### 8. 文件名处理

标题超 200 字符会截断。特殊字符（`<>:"/\|?*`）替换为 `_`。

### 9. 默认只读

默认不修改源缓存。例外：解密临时文件（自动清理）和 `--delete-source`。

## 缓存目录在哪里

### Android

```
/sdcard/Android/data/tv.danmaku.bili/download/
```

不同版本：
- 国内版：`tv.danmaku.bili`
- 概念版：`com.bilibili.app.blue`
- 谷歌版：`com.bilibili.app.in`
- HD版：`tv.danmaku.bilibilihd`

需要先拷贝到电脑再使用。

### Windows

```
C:\Users\<用户名>\Videos\bilibili\
```

或在 B站客户端设置中查看"下载"路径。

### Mac

```
~/Library/Containers/com.bilibili.bili/Data/Documents/
```

## 常见问题

### Q: 提示找不到 ffmpeg

确保 `ffmpeg.exe`（Windows）或 `ffmpeg`（Linux）与脚本在同一目录，或用 `--ffmpeg` 指定路径。Linux 可 `apt install ffmpeg`。

### Q: 合并后视频没有声音

Windows/Mac 缓存中如果两个 m4s 文件都不以 `30280.m4s` 结尾，脚本按文件大小判断（小的是音频）。极端情况下可能判断错误。

### Q: 终端显示乱码

Windows 终端默认 GBK 编码，运行前执行 `chcp 65001` 切换到 UTF-8。

### Q: 同一个视频缓存了多个画质

脚本自动选择最高画质。Android 缓存中每个视频可能有多个画质子目录（如 `64/`=720P、`80/`=1080P），脚本选择画质ID最大且文件完整的目录。

### Q: 扫描到的视频数量比预期少

1. 检查 `--platform` 参数是否正确
2. 确认缓存目录路径是包含数字编号子文件夹的那一层
3. 检查终端 `[警告]` 信息是否有权限被跳过的目录

### Q: 可以同时运行多个实例吗

可以，只要输出目录不同。相同输出目录下有文件重名检测，不会覆盖但可能产生混乱编号。

### Q: 如何只合并某几个视频

将需要的缓存文件夹单独拷贝到一个新目录，然后指向该目录运行。
